Quote Generation & Approval System Design

This is one project I worked on, in conjunction with my Master's program capstone. The description of the project that follows serves as one example among the many other similar projects I have undertaken during my time here at Tokyo Electron and, hopefully, illustrates my role as an intermediary between the people and the technology for better facilitating the former's business needs.

Introduction

Tokyo Electron is a company that manufactures semiconductor equipment. It has a major customer who request quotes on a regular basis, whether it is to procure new tools to modify any already-owned tools. The process of generating these quotes is specific to this customer and goes through a complicated set of procedures.

The Problem

Once a quote request is placed it goes through a system in which the quote is created and then passed through several approval stages via automated notifications and approval tracking. Hosted on SharePoint, the system was built with InfoPath and SharePoint Designer. But Microsoft is dropping support for InfoPath in several years.

Another Problem

This system is used exclusively by one business unit (BU 1). But another business unit (BU 2) has requested setting up its own automated system, similar to the one BU 1 is using. Currently, the quote requests BU 2 receives from the customer are created and approved manually, which makes tracking multiple quotes difficult.

Goals

To produce a business solution that 1) allows BU 1 to continue generating quotes and gather approvals in an automated manner using new mathods and developer tools and 2) provides BU 2 with an automated system of its own. The new system should be funtional, user-friendly, and maintainable.

Interviewing

Acting as the project manager and main developer, I gathered requirements from key stakeholders—individuals who create quotes as well as those who approve them. The system comprises of many parts—I needed to understand its various components and how they all worked together before proposing any solution.

Assessing

I analyzed the available developer tools that could replace InfoPath and SharePoint Designer and offered four different options on how to build the system. Each option had its pros and cons, and I made my recommendation to the team and stakeholders, who subsequently selected one option out of the four.

Developing

I began to develop the new system in a test environment using a combination of SharePoint Server, Microsoft Excel, and Nintex Workflow. The Excel files required macros, which in turn led me down the road of having to learn and implement basic VBA. Stakeholders provided feedback on a weekly basis through meetings, where I showed project progress by giving piecemeal demonstrations on what I had been able to build.

Testing

Functional testing and usability testing took place to confirm the new system worked the way it was meant to, and to ensure users did not have troubles using it. The new system continued to be refined over the course of the next several months, after launching in a production environment.

Results

I finished creating a quote generation and approval system for both BU 1 and BU 2. It operates as follows: The quote is created using an Excel file, the template of which is automatically generated via a Nintex workflow. A quote number is also auto-generated at the same time. After the quote is filled out and submitted for approval, a separate workflow reads the file and processes it for approval by sending it to the appropriate approvers in three sequential stages. After the quote receives all approvals it is published for customer review.

Challenges

Rebuilding the system for BU 1 was relatively smooth but progress had been slower translating the manual quote process into an automated one for BU 2. Initial idea was to "copy" BU 1's new system for BU 2 but business processes and user history/expectations were different and had to be accounted for, in order to execute proper change management. System launch, training, and documentations were delayed but eventually completed.

TRAFFIC SCORING

Measuring Traffic with Google Apps API

Problem statement

Poor commute is a common experience in large cities. As drivers we want to understand the general flow of traffic, but traffic sensor data is inaccessible to the public. Is there a simple way to quantify, analyze, and discover traffic patterns by ourselves?

Idea #1: Google Maps

Step 1: Open map of Austin in browser

Step 2: Take a screenshot at set intervals using auto screenshot capture + auto refresh applications

Step 3: Count the number of green, orange, red, and maroon pixels using Python script

Step 4: Generate traffic scores

But...

Issues

We cannot use PC while running setup. It is inefficient to save images and process them one by one. Inconsistencies become a common theme from trying to sync up all applications, and manual setup sometimes leads to human error.

Idea #2: Google Maps Distance Matrix API

We can automatically collect travel time from various routes using the Distance Matrix API. After the script for it is written, setup is easy: Set timer, start script, let it run in the background, and data is written in csv file (instead of as large png files).

We collect data over a span of a few weeks, at which point we are then able to convert times into traffic severity scores and aggregate scores.

Results

Austin Traffic Score App

We have created a simple web app to display general traffic patterns in Austin, divided by main regions. It uses Google Maps API + collected historical data to "predict" future traffic severity of a region.

Github: Austin Traffic

Limitations

Data needs to be validated with actual sensor data.

Route selection and region mapping bias.

Only two weeks' worth of data collected.

Conclusion

There is potential for method to be used in future traffic studies. Google Maps API is inexpensive, accessible, and scalable. The information is also useful on personal level, informing user on traffic patterns within own commute

BROWNIE

HOME AUTOMATION FOR THE ELDERLY

Brownie is a concept for a smart home application for senior citizens. By surrounding them with an integrated system that monitors many of their household, health, and security needs, senior citizens who live on their own can be assured of greater independence in their own homes, thereby reducing their need for caretaker support.

Problem statement

Senior citizens are projected to increasingly live on their own. They may require the support of their children or caretakers, the latter of which may:

  • • Be expensive to hire
  • • Be difficult to find
  • • Deprive the elderly of a sense of independence

Our idea

We conceived a comprehensive smart home system catered for the elderly.

Challenges

Such a system would need to tackle the following challenges posed by the target audience, to name a few:

  • • Limited mobility
  • • Physical disabilities or impairment
  • • Discomfort with technology
  • • Emergencies such as falling accidents

Design process

Competitor Analysis

We cross-examined related companies and products:

HoneyCo | Samsung SmartThings | Crestron | Lively Activity Sensors

Our purpose was to identify their strengths and weaknesses and find areas for opportunity. HoneyCo came the closest to paralleling what we trying to accomplish but lacked user-friendliness and comprehensiveness.

Contextual Inquiry

To design a smart home system that best served the needs of the older population, we had to understand the following:

  • • How they lived their daily lives and what difficulties they faced
  • • Their health conditions and problems
  • • Their interactions with and attitude toward high-tech devices
  • • How their adult children played a role

Interviews

Three older adults between the ages of 61 and 70, as well as three younger adults between the ages of 43 and 46, were interviewed. Seniors were generally less comfortable with technology, but privacy was considered very important by all. Additionally, health was seen as major concern. For the most part there was frequent communication between parents and children.

Survey

We conducted an online survey for adults 35 and up. Respondents who were 55+ reported lower trust level for automation and lower comfort with technology. Most respondents valued privacy, security, safety, health monitoring.

Opportunity

Demand for automation services was low among elderly but could be ascribed to unfamiliarity with technology in general. Our aim was to to be able to bridge this gap by creating a user-friendly and useful solution for seniors.

Affinity Diagram


Main points from interviews and surveys were collated, then sorted into categories: Automation, Reminder, Health, Contact, and Other.

Personas

We created personas, who embodied the voice of our interview and survey participants, to help guide our design process.

Frank: A 68-year-old who needs a solution for tackling his forgetfulness and bonding with his family.

Vickie: A 42-year-old who wants to have her mother taken care of with the assistance of technology.

Ideation

Using the ideas and concepts we gathered up to this point, we began to visualize them in the form of a mobile app.

  • • Single interface: All user interaction would begin and end at the app
  • • Maximizing ease of use for seniors where manual interactions are needed
  • • Automating everything else

Sketches

Each team member drew wireframe sketches on paper individually.

Interface designs then drawn on a whiteboard to discuss everyone's ideas.

Low fidelity prototype

All of our prototypes were built with Sketch and InVision. The architecture of the app was based on categories from affinity diagram. Each category serves a specific function, including automating the home, monitoring health status, scheduling reminders, and allowing direct contact with family or emergency services.

User testing 1

We conducted face-to-face usability tests with eight participants. We wrote a script consisting of five basic tasks. Unfortunately, lack of resources led to the recruitment of young participants for the first test round (all under 30). We were nonetheless able to gather plenty of useful feedback on the app.

Modification 1

Contact function was relegated in favor of a Help section due to it being redundant with a smartphone's native features and with social media apps. Wording was changed to enhance clarity. We narrowed the scope of the reminder system to medications and reduced number of navigatable screens. We also introduced color coding to distinguish normal from abnormal health status.

High fidelity prototype

User testing 2

Usability tests were conducted with twelve remote participants from UserTesting.com. Unlike the first round of testing, participants were closer to our target user group, between the ages of 55 and 70.

Results

Participants rated each task and the app overall.

Modification 2

Feedback was very positive: Participants liked the simplicity and intuitiveness of an app that handles various household functions and monitoring. This aligned with our goal to keep the system simple for those less comfortable with technology, but there were a few small suggestions for change that we implemented:

  • • Icon readability was improved, particularly on the health screen
  • • Vocabulary was modified for better comprehension

High fidelity prototype 2

Lessons Learned

Thinking from the perspectives of senior citizens meant designing an interface a little differently from how we otherwise might have designed it. For example, it was important to use bigger icons and bigger fonts than usual, to avoid the use of "hidden" screens, and to employ high contrasts to enable better readability.

  • • To increase traffic
  • • To increase awareness
